This is similar to rbslapd1.rb but here we use TOMITA Masahiro’s prefork library: <raa.ruby-lang.org/project/prefork/> Advantages over Ruby threading:

  • each client connection is handled in its own process; don’t need to worry about Ruby thread blocking (except if one client issues overlapping LDAP operations down the same connection, which is uncommon)
  • better scalability on multi-processor systems
  • better scalability on single-processor systems (e.g. shouldn’t hit max FDs per process limit)

Disadvantages:

  • client connections can’t share state in RAM. So our shared directory now has to be read from disk, and flushed to disk after every update.

Additionally, I have added schema support. An LDAP v3 client can query the schema remotely, and adds/modifies have data validated.
